# Shakespeare-Aware RAG System

This project implements a small, runnable Retrieval-Augmented Generation system for CSCI433/933 Assignment 2. It supports beginner-friendly question answering over three required Shakespeare plays:

- Hamlet
- Macbeth
- Romeo and Juliet

The system loads the structured play dataset, creates retrieval chunks, builds a cached retrieval index, retrieves relevant evidence for a user question, and generates an answer that is shown together with the retrieved passages.

## Project Structure

```text
data/
  raw/                         Provided structured Shakespeare dataset
prompts/
  system_prompt.txt            RAG behaviour instructions
  stylised_prompt.txt          Creative style-generation instructions
results/
  instructor_questions.json    Evaluation questions
  chroma_db/                   Persistent ChromaDB vector store
  chroma_retriever.json        Saved Chroma retriever metadata
  evaluation_results.csv       Baseline and RAG outputs, generated by evaluate.py
src/
  config.py                    Paths and retrieval settings
  data_loader.py               Loads the three required plays
  chunking.py                  Creates overlapping retrieval chunks
  retrieval.py                 ChromaDB retriever
  baseline.py                  Simple baseline retrieval system
  build_index.py               Builds and tests the retrieval index
  rag_chatbot.py               Interactive RAG chatbot
  evaluate.py                  Runs baseline and RAG on evaluation questions
```

## Setup

Install the required packages:

```bash
pip install -r requirements.txt
```

The retrieval backend is ChromaDB with `sentence-transformers/all-MiniLM-L6-v2` embeddings. ChromaDB is required; the RAG system raises an error if ChromaDB is unavailable instead of falling back to another retriever.

## Dataset

The code expects the provided dataset files in:

```text
data/raw/
  hamlet.json
  macbeth.json
  romeo_and_juliet.json
```

The loader reads the structured scene records from these files. Each record preserves useful metadata including play, act, scene, scene summary, keywords, and source text.

## Chunking Strategy

The system uses scene-level records and splits long scenes into overlapping word windows. This keeps the evidence local enough for retrieval while preserving more context than isolated single utterances. Each chunk keeps metadata for display and evaluation.

Default settings are in `src/config.py`:

```python
CHUNK_WORDS = 260
CHUNK_OVERLAP = 45
DEFAULT_TOP_K = 3
```

## Build the Retrieval Index

Run:

```bash
python src/build_index.py
```

This loads all three plays, creates chunks, stores vectors in ChromaDB, saves retriever metadata to `results/chroma_retriever.json`, and prints a retrieval test for:

```text
Why does Macbeth kill Duncan?
```

## Run the Chatbot

Run:

```bash
python src/rag_chatbot.py
```

Example questions:

```text
Who is Hamlet?
Why does Macbeth kill Duncan?
What is the conflict between the Montagues and the Capulets?
Why does Hamlet delay taking revenge?
Generate a short Shakespearean-style response from Juliet explaining her conflict after meeting Romeo.
```

For each question, the chatbot prints:

- retrieved evidence with play, act, scene, speaker, and excerpt;
- the generated answer;
- a preview of the RAG prompt used to condition generation.

Stylised-response detection uses semantic similarity between the user prompt and style-intent phrases, rather than direct keyword matching. The same embedding model is also used to select the closest stylised topic from the prompt and retrieved evidence.

Stylised generation uses `prompts/stylised_prompt.txt`, which keeps creative responses under 150 words and marks them as creative output rather than factual evidence.

## Language Model Interface

The system includes an optional local SLM interface through Ollama. If `OLLAMA_MODEL` is set, `rag_chatbot.py` sends the RAG prompt to the local Ollama API.

Example:

```bash
ollama pull llama3.2:3b
set OLLAMA_MODEL=llama3.2:3b
python src/rag_chatbot.py
```

On PowerShell:

```powershell
$env:OLLAMA_MODEL="llama3.2:3b"
python src\rag_chatbot.py
```

If Ollama is not running or no model is configured, the chatbot uses a compact local fallback generator. The fallback is useful for demonstration and testing, but the report should clearly state which generation mode was used for final evaluation.

## Baseline System

`src/baseline.py` implements a simple comparison baseline using the same ChromaDB retriever as the RAG chatbot. It retrieves only the single most relevant passage and returns it without synthesising a beginner-friendly explanation.

The improved system in `src/rag_chatbot.py` retrieves multiple passages from ChromaDB, builds a RAG prompt, and generates a beginner-friendly answer conditioned on those passages.

## Evaluation

Run:

```bash
python src/evaluate.py
```

This writes:

```text
results/evaluation_results.csv
```

The CSV contains baseline and RAG outputs for the instructor questions. Students should manually complete the scoring columns:

- correctness score
- grounding score
- retrieval relevance score
- usefulness score
- style quality score
- comments

Use a consistent scale, such as 1 to 5, and include both successful examples and failure cases in the report.

## Error Analysis

The final report should discuss observed weaknesses such as:

- retrieval returning a nearby scene but missing the best motive passage;
- answers relying too heavily on scene summaries;
- stylised output becoming less precise than factual explanation;
- ambiguous questions retrieving evidence from the wrong play or scene;
- fallback generation being less flexible than a real local SLM.

## Reproducibility Notes

The system is designed to run on a standard laptop. It does not train a model from scratch and does not require a GPU. ChromaDB vectors are cached in `results/chroma_db`, so the index does not need to be rebuilt during demonstration unless the dataset or chunking settings change.
